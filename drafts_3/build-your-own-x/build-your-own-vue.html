前言

这是 build your own x 的 vue 版,想做一个mini 版 vue 的主要原因是最近需要修改老项目的前端,被以前的 jquery 各种 DOM 操作,HTML 拼接折磨地心力交瘁,新模块新页面不需要 build,直接引入 vue就可以写,感受到了数据驱动对程序员生活的改善

数据驱动 (也叫做响应式 Reactivity)

以前如果数据发生了变化,每次都需要在 DOM 树上先找到对应的 DOM 对象, 然后写代码对其进行修改 , 这就造成了业务代码上视图(View)和数据(Model)的强耦合

MVVM 的模型出现后, Model 和 View 之间出现了一层 ViewModel , 将他们解耦开来

VM 层负责的工作就是,如果数据发生变化了, 触发与该数据绑定的 DOM 监听程序, 该程序会负责修改视图
如果 DOM 发生变化了, 触发与该 DOM 相关的数据监听器, 修改对应数据

实现数据监听的工具是 JS 引擎提供的 definedProperty

实现 DOM 监听的工具是? addEventListener

那么 Vue 是如何实现相关的事件监听过程的呢? 这就涉及到编译原理了…

举例 <h></h> , 当Compiler 扫描到此处时,发现是绑定在h DOM 对象下的 text ,
此时就会关联一个该 DOM 对象 text 变化的监听器,修改对应数据
同时也会关联一个该数据变化的监听器,修改对应 DOM 对象的text 内容

可以看到 ViewModel 帮我们把脏活累活都干了,大部分复杂的监听程序都通过Compiler 扫描后生成,工作量并不小,

这个 mini 版实现的是一个写死的监听器,仅支持监听 text 内容变化,不过足够理解原理了

Virtual Dom

为什么会有 Virtual Dom 这个东西呢

1.每次对真实 DOM 的修改都会触发浏览器的渲染和重绘 , VDOM 的出现减少了浏览器渲染和重绘的次数 (对树进行 DIFF)

2.VDOM 不和浏览器环境绑定,因此业务逻辑代码可以很容易地移植到其他环境,比如 Native APP , PC 程序(Electron)

或许还有其他好处,暂时只知道这两个,所以如果只是想解耦数据和视图,没有 VDOM 也是可以的! 但是一般的框架都会有 VDOM 来实现更多复杂的功能,
因此变化的方向就变成了 数据变化 -> VOM -> 真实 DOM

VDOM 其实就是一个由JavaScript 对象组成的树,和 DOM树的节点 一一对应
